# Gemini Chrome Agent — Agent Guide

This file contains everything an AI coding agent needs to know to work effectively on the **Gemini Chrome Agent** project.

## Project Overview

The Gemini Chrome Agent is a Chrome/Brave browser extension (Manifest V3) paired with a Python native messaging host. It bridges the Google Gemini web interface (`gemini.google.com`) directly to the local Ubuntu environment, enabling Gemini to act as an autonomous agent that can run shell commands, read/write/list local files, run git and Python, and attach local files to the chat (see **Supported Actions** for the full set of eight).

When Gemini generates a JSON code block containing one of these actions, the extension detects it, executes it locally, and injects the result back into the Gemini chat input box — creating a closed autonomous loop.

## Technology Stack

| Layer | Technology | Files |
|-------|-----------|-------|
| Browser Extension | TypeScript, WXT (Manifest V3) | `entrypoints/background.ts`, `entrypoints/content.ts`, `entrypoints/popup/` |
| Native Messaging Host | Python 3 (stdlib only) | `host.py`, `main.py` |
| Setup & Tooling | Bash, Node.js, npm | `setup.sh`, `package.json`, `wxt.config.ts` |
| Optional Log Server | Python 3 (stdlib only) | `log_server.py` |

The extension is built with [WXT](https://wxt.dev/) and loaded as an unpacked extension from the build output directory. TypeScript source lives in `entrypoints/` and `utils/`; WXT compiles it to `.output/chrome-mv3/`.

## Architecture

The system has three layers:

1. **Frontend (`content.ts`)** — Injected into `gemini.google.com`. Uses a `MutationObserver` to scan the DOM for `<pre><code>` blocks containing JSON payloads with an `"action"` key. Validates, deduplicates, and forwards payloads to the background script. Receives responses and injects them into the Gemini input box, then auto-clicks the Send button. A second MAIN-world content script (`entrypoints/uploader.content.ts`) handles `attach_files` by injecting `File` objects into Gemini's uploader (see **Gemini DOM facts**).

> **Gemini DOM facts (verified live; the un-unit-testable layer — Google changes this without warning).** The composer is a **Quill editor (Angular Material), not React.** **Inject text by writing Quill's model with the `'user'` source: `quill.setText(text, 'user')` (then `setSelection`).** On the current `new-input-ui` composer (as of 2026-06), `execCommand('insertText')` updates the DOM but **desyncs Quill's model**, and `quill.setText` with the default `'api'` source syncs the model but leaves **Send disabled** — only a `'user'`-source text-change arms Send. The content script's **isolated world cannot reach `__quill`**, so `injectText` posts the text to the **MAIN-world bridge** (`entrypoints/uploader.content.ts`, message `{__gla:'quill-set'}`), which calls the Quill API; `triggerSend()`'s synthetic click then submits. Send readiness is gated by **`pointer-events`/`disabled`** (don't trust `disabled` alone). File upload works by hooking `HTMLInputElement.prototype.click` from the MAIN world and feeding a `DataTransfer`; synthetic paste and drag-drop do **not** work. All of this lives in `utils/injection.ts` and `entrypoints/uploader.content.ts`, and must be re-verified against live Gemini (not unit tests) when Google changes the UI (see #16 for the `new-input-ui` break).

2. **Middleware (`background.ts` + Native Messaging)** — A Manifest V3 service worker that maintains a `browser.runtime.connectNative` port to `com.local.gemini_agent`. Routes messages between `content.ts` and `host.py`. Uses `browser.storage.session` to persist the sender tab ID so responses can be routed correctly even after the service worker sleeps and wakes up.

3. **Backend (`host.py`)** — A persistent Python script that communicates over stdin/stdout using Chrome's length-prefixed JSON native messaging protocol (`struct.pack('@I', len)`). Executes shell commands via `subprocess.run`, reads/writes files, and returns JSON responses.

## Supported Actions

All actions are represented as JSON code blocks inside the Gemini chat. The content script recognizes eight actions (kept in sync across `isValidPayload()` in `utils/protocol.ts`, `VALID_ACTIONS` in `utils/config.ts`, the `AgentAction` type in `utils/types.ts`, and the dispatch in `host.py`):

| Action | Required Fields | Behavior |
|--------|----------------|----------|
| `run_shell` | `command` | Executes the command via `subprocess.run(command, shell=True, timeout=30)`. Returns stdout, stderr, and exit code. |
| `write_file` | `filepath`, `content` | Writes `content` to `filepath` (creates directories as needed). Overwrites existing files. |
| `read_file` | `filepath` | Reads the entire file and returns its contents. Returns an error if the file does not exist. |
| `list_files` | `filepath` | Lists directory entries (or notes the path is a file / not found). |
| `git_status` / `git_diff` | `filepath` (dir, defaults `.`) | Runs `git status` / `git diff` in the directory. |
| `run_python` | `content` or `filepath` | Runs inline code (temp file) or an existing script via `sys.executable`. |
| `attach_files` | `filepaths` (non-empty array); optional `prompt` | Reads each file in `host.py`, streams it to the extension as size-guarded base64 chunks, and a MAIN-world script injects them into Gemini's uploader. With a `prompt` (or a default summary) it then honors the auto-submit toggle. |

## Agent Skills (cross-harness)

This repo ships reusable [agent skills](https://moonshotai.github.io/kimi-cli/en/customization/skills.html) (`SKILL.md` bundles) that must be discoverable from **any** AI coding harness used on the project — Claude Code, Kimi (kimi-code / kimi-cli), Codex, etc. Each harness scans a *different* directory, so the layout is **one canonical store + per-harness symlinks**, mirroring this host's global `~/.agents/skills/` convention.

**Canonical store:** `.agents/skills/<skill-name>/` holds the real files. `.agents/skills` is the *generic* skills directory that Kimi (both kimi-code and kimi-cli ≥ 1.39) auto-discovers natively by walking up to the nearest `.git` ancestor — **no symlink needed for Kimi.** Only Claude Code needs a symlink, because it scans `.claude/skills/` exclusively (project scope) and has no setting to add a directory.

| Harness | Scans (project scope) | How it sees the skill |
|---------|-----------------------|-----------------------|
| Kimi (kimi-code) | `.agents/skills`, `.kimi-code/skills` | **native** — reads the canonical store directly |
| Kimi (kimi-cli ≥ 1.39) | `.agents/skills`, `.kimi/skills`, `.claude/skills`, `.codex/skills` | **native** via `.agents/skills` |
| Claude Code | `.claude/skills` only | via the `.claude/skills/<name>` symlink → `../../.agents/skills/<name>` (Claude Code follows symlinks at discovery + load time) |

Skill discovery resolves real paths (`realpath`), so the symlink is followed transparently. Git stores it as mode `120000` (the relative target string), not duplicated content. Empirically verified 2026-05-31: kimi-code 0.6.0 and kimi-cli 1.45.0 both load `chrome-extension-builder` from `.agents/skills`, and Claude Code loads it via the `.claude/skills` symlink.

**Adding a new skill:** create it under `.agents/skills/<name>/` (with a `SKILL.md` whose frontmatter has at least `name` and `description` — the only universally-required fields), then add the Claude Code symlink:
```bash
ln -s ../../.agents/skills/<name> .claude/skills/<name>
git add .agents/skills/<name> .claude/skills/<name>
```

**Supporting a new harness:** if it reads `.agents/skills` natively (Kimi, Codex), nothing is needed. Otherwise add one symlink from its project skills dir to `.agents/skills/<name>` — e.g. a pre-1.39 kimi-cli would need `ln -s ../../.agents/skills/<name> .kimi/skills/<name>`.

## Setup & Installation

The extension is built with WXT and loaded as an unpacked extension from the build output:

1. Install dependencies and build the extension:
   ```bash
   npm install
   npm run build
   ```
   For development with hot-reload, use `npm run dev` instead.
2. Open Brave/Chrome and navigate to `brave://extensions/` (or `chrome://extensions/`).
3. Enable **Developer mode**.
4. Click **Load unpacked** and select `.output/chrome-mv3/` inside this project directory.
5. Copy the generated **Extension ID**.
6. Run `./setup.sh` and paste the Extension ID when prompted.
7. The script will:
   - Make `host.py` executable
   - Generate `com.local.gemini_agent.json` with absolute paths and the Extension ID
   - Create the Brave Native Messaging directory at `~/.config/BraveSoftware/Brave-Browser/NativeMessagingHosts/`
   - Symlink the manifest into that directory
8. Refresh the Gemini tab.

**Order matters:** the extension must be loaded first (to generate the Extension ID) before running `setup.sh`, because the Native Messaging manifest must whitelist that specific Extension ID.

### Live debugging against real Gemini

`./scripts/launch-debug-brave.sh` launches a dedicated-profile Brave with the extension
loaded and CDP on port 9222. **Critical:** it loads from the absolute path
`.output/chrome-mv3` with **no trailing slash** — that exact path hashes to the extension
ID `pggfgfolmomlhlnabkfboiiojdkhapoh`, which `com.local.gemini_agent.json` is locked to. A
different path (or a trailing slash) yields a different ID and silently breaks native messaging.

`.mcp.json` defines a `brave-debug` MCP server (chrome-devtools-mcp) that attaches to that
Brave over CDP via `--browserUrl http://127.0.0.1:9222`. Any MCP-capable client (Claude Code,
Cursor, etc.) can then drive the live page — typical flow: `list_pages` → `navigate_page` →
`evaluate_script` / `take_snapshot`. Launch the script first; the client picks up the MCP
server per its own config-load rules (e.g. Claude Code loads `.mcp.json` at session start, so
restart the session). Use this to verify the Gemini DOM layer (injection, send, upload) against
the real page rather than trusting selectors — Google changes the UI without warning.

## Code Style Guidelines

### TypeScript (`background.ts`, `content.ts`, `utils/*.ts`)

- Use `let`/`const`; avoid `var`.
- Prefix all console logs with a domain tag, e.g. `[Background]`, `[Gemini Agent]`.
- Use semi-colons.
- Use `try/catch` liberally around DOM queries and messaging calls to prevent uncaught promise rejections.
- Service-worker code must be resilient to sleep/wake cycles: persist state in `browser.storage.session`, not in memory.
- Prefer explicit types for function parameters and return values.

### Python (`host.py`, `main.py`, `log_server.py`)

- Use snake_case for functions and variables.
- Use type hints where appropriate (see `main.py` for examples).
- Include docstrings for public functions.
- Log to `/tmp/gemini_host.log` using the standard `logging` module.
- Truncate large command/file *output* (1MB cap via `truncate_output` in `host.py`) to prevent breaking the native messaging protocol. **Exception:** `attach_files` must NOT truncate — it instead streams base64 in 512KB chunks with a 25MB per-file guard (`ATTACH_CHUNK_SIZE` / `ATTACH_MAX_BYTES`), reassembled by `utils/attach.ts`.

## Safety & Anti-Looping Mechanisms

The `content.ts` script contains several guards to prevent runaway execution:

| Guard | Value | Purpose |
|-------|-------|---------|
| Cooldown | 15 seconds (`COOLDOWN_MS = 15000`) | Prevents immediate re-execution after a payload is handled. |
| Rate limit | 5 per minute (`MAX_PER_MINUTE = 5`) | Auto-pauses the agent if exceeded. |
| Settling period | 5 seconds (`SETTLING_PERIOD_MS = 5000`) | Ignores payloads present at page load to avoid re-processing historical blocks. |
| Deduplication | 60-second TTL (`PAYLOAD_TTL_MS = 60000`) | Tracks processed payloads by hash to avoid duplicate execution. |
| Valid actions whitelist | `run_shell`, `write_file`, `read_file`, `list_files`, `git_status`, `git_diff`, `run_python`, `attach_files` | Only these eight actions are accepted; all others are ignored. |
| Manual pause | `Alt+Shift+K` or on-screen button | User can pause/resume the agent at any time. |

When modifying `content.ts`, **preserve these guards**. Changing thresholds or removing checks can cause dangerous recursive loops where Gemini's own output triggers new actions indefinitely.

## Security Considerations

- **Local execution:** `host.py` runs shell commands with the same privileges as the user running the browser. Any command Gemini generates is executed directly.
- **Origin lock:** The Native Messaging manifest (`com.local.gemini_agent.json`) restricts communication to a single Extension ID.
- **Path expansion:** `host.py` expands `~` via `os.path.expanduser` for `read_file` and `write_file`.
- **No sandboxing:** There is no privilege separation, containerization, or command allow-list. This tool is intended for personal, trusted use only.

## Common Issues

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| `Specified native messaging host not found` | Symlink missing or Extension ID mismatch | Re-run `setup.sh` with the correct Extension ID. |
| `Could not establish connection. Receiving end does not exist.` | `content.js` not injected | Refresh the Gemini tab after reloading the extension. |
| Service Worker shows "Inactive" | Normal Manifest V3 sleep behavior | Click the service worker link to wake it up. |
| Empty "System Result" | Command produced no output | `host.py` now returns a placeholder message when stdout+stderr are empty. |

## Modifying the Project

- **Adding a new action:** Update the `AgentAction` type in `utils/types.ts`, `isValidPayload()` in `utils/protocol.ts`, `VALID_ACTIONS` in `utils/config.ts`, add the handler + dispatch branch in `host.py`, and mirror it in `main.py` for CLI testing. Keep all four sources in sync.
- **Changing Gemini page selectors:** If Google updates their DOM, update the selectors in `injectText()` / `triggerSend()` in `utils/injection.ts` and the uploader button/menu lookups in `entrypoints/uploader.content.ts`. Re-verify live (see **Gemini DOM facts**) — Quill `execCommand` injection and `pointer-events`-gated send are easy to get wrong against unit-test mocks.
- **Changing the host name:** If you rename `com.local.gemini_agent`, update it in `wxt.config.ts` (runtime config), `entrypoints/background.ts`, and the filename of the host manifest JSON.
